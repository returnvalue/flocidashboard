{% if service.key == 'acm' %}
About Floci ACM

Local ACM stores AWS-compatible certificate metadata and issues requested certificates immediately for local emulation.

ACM is most often used to test TLS certificate wiring for load balancers, CloudFront, API endpoints, and infrastructure automation. In Floci, a good learning flow is to request a certificate, inspect the issued metadata and validation options, add tags, then export a private certificate when testing workflows that need key material.

Certificates use real RSA or EC keys and valid X.509 structure. Validation is emulator-friendly, and only PRIVATE certificates can be exported with an encrypted private key.

{% elif service.key == 's3' %}
About Floci S3

Local S3 emulator. Bucket policies and lifecycle rules are stored but not enforced at access time. Replication, access logging, and inventory are not available.

S3 is most often used as durable object storage for uploads, generated files, backups, static assets, data lake inputs, and service-to-service handoffs. In Floci, a good learning flow is to create a bucket, upload a file, organize objects with folder-style prefixes, inspect metadata and tags, then generate a presigned URL to test temporary access.

Common local test cases include upload and download paths, overwrite behavior, empty bucket cleanup, versioning changes, object tagging, copy operations, prefix browsing, and app behavior when a bucket or object does not exist.

{% elif service.key == 'iam' %}
About Floci IAM

Local IAM stores users, roles, groups, policies, access keys, and trust relationships for AWS-compatible development workflows.

IAM is most often used to shape what local SDK calls are allowed to do and to test role-based credentials. In Floci, a good learning flow is to inspect a principal, review its attached and inline policies, create a temporary access key or assume a role, then test those credentials from your app.

Common local test cases include missing permissions, incorrect trust policies, inactive access keys, group policy inheritance, permission boundaries, and role-session credential wiring.

{% elif service.key == 'sqs' %}
About Floci SQS

Local SQS emulator for queue-driven workflows. Polling messages makes them temporarily invisible until the visibility timeout expires or they are deleted.

SQS is most often used to decouple producers from background workers, smooth traffic spikes, retry failed work, and connect services without requiring them to run at the same time. In Floci, a good learning flow is to create a queue, send a test message, poll it from the UI, inspect the body and attributes, then delete it after your worker logic succeeds.

Common local test cases include producer message format, consumer retry behavior, visibility timeout handling, delete-after-success behavior, purge cleanup, FIFO ordering, duplicate handling, and dead-letter queue flows.

{% elif service.key == 'ses' %}
About Floci SES

Local SES supports the classic Query API and a subset of SES v2 REST JSON. Identity verification succeeds immediately, so test senders and recipients can be created without DNS or inbox verification.

Emails are always captured locally at /_aws/ses, even when SMTP relay is configured. The workbench focuses on verifying identities, sending sample messages, managing templates, and inspecting or clearing the captured mailbox.

SES v1 and v2 share identity, template, configuration set, and sent-message state in Floci. Configuration set event destinations are stored locally, with SNS destinations used for emitted SES events when configured.

{% elif service.key == 'sns' %}
About Floci SNS

Local SNS emulator for topic-based pub/sub workflows. Published messages are delivered immediately to supported local subscriptions, including SQS queues.

SNS is most often used to fan out events to queues, functions, webhooks, and notification handlers. In Floci, a good learning flow is to create a topic with your app or CLI, subscribe an SQS queue, publish a test event here, then inspect the delivered message in the SQS workbench.

Common local test cases include message shape, attributes, subject handling, subscription filter policies, FIFO group and deduplication behavior, and SNS-to-SQS fanout.

{% elif service.key == 'lambda' %}
About Floci Lambda

Local Lambda runs function code in Docker-backed execution environments. Invocations can return response payloads, function errors, and a tail of execution logs.

Lambda is most often used to handle API Gateway requests, consume SQS messages, react to S3 or EventBridge events, and run small background tasks. In Floci, a good learning flow is to select a function, invoke it with representative JSON, inspect the payload and log tail, then follow the related CloudWatch log group.

Common local test cases include handler payload parsing, environment variables, timeout behavior, function errors, event-source integration payloads, and log output.

{% elif service.key == 'kms' %}
About Floci KMS

Local KMS stores symmetric and asymmetric keys, aliases, tags, policies, rotation state, deletion state, and grant records through the AWS-compatible JSON 1.1 API.

KMS is most often used by apps and services to encrypt data, test alias wiring, generate envelope-encryption data keys, and validate key lifecycle paths. In Floci, a good learning flow is to create a symmetric key, add an alias, encrypt a sample payload, decrypt the ciphertext, then try a generated data key.

Grant lifecycle APIs are stored and queryable in Floci, but grants are not evaluated for cryptographic authorization. The reserved creation-time tag floci:override-id can make deterministic local key IDs for tests.

{% elif service.key == 'dynamodb' %}
About Floci DynamoDB

Local DynamoDB stores application state in tables with AWS-compatible key, index, stream, and item APIs. PartiQL SELECT support is available in Floci 1.5.17 and newer.

DynamoDB is most often used for low-latency application records, counters, job state, idempotency keys, and event projections. In Floci, a good learning flow is to inspect a table schema, scan a bounded set of items, select an item for JSON detail, then try a read-only PartiQL SELECT query.

Common local test cases include key design, item shape, stream enablement, TTL configuration, index projections, and app behavior when records are missing or malformed.

{% elif service.key == 'cloudwatch' %}
About Floci CloudWatch Logs

Local CloudWatch Logs captures log groups, log streams, and log events produced by services such as Lambda. This viewer focuses on recent logs for local debugging rather than the full CloudWatch Console surface.

A good learning flow is to select a Lambda log group, open the newest stream, refresh recent events, and compare the messages with the request or event that triggered the function.

Common local test cases include handler errors, async event processing, retry behavior, missing environment configuration, and log messages emitted by background workers.

{% elif service.key == 'stepfunctions' %}
About Floci Step Functions

Local Step Functions runs state machine workflows through the AWS-compatible Amazon States API. This workbench focuses on starting executions and inspecting their status, input, output, and recent history.

Step Functions is most often used to coordinate Lambda functions, service integrations, retries, waits, and branching workflows. In Floci, a good learning flow is to select a state machine, start an execution with representative JSON, inspect the event history, then stop a running execution when testing cancellation behavior.

Common local test cases include input shape, Choice state routing, Lambda task payloads, retry and catch paths, timeout behavior, failure output, and task-token callback flows.

{% elif service.key == 'cloudformation' %}
About Floci CloudFormation

Local CloudFormation deploys templates through the AWS-compatible Query XML API and provisions supported local resources such as S3 buckets, SQS queues, SNS topics, DynamoDB tables, Lambda functions, IAM identities, SSM parameters, KMS keys, Secrets Manager secrets, API Gateway resources, EventBridge rules, and more.

CloudFormation is most useful when testing whole local application slices. In Floci, a good learning flow is to validate a template, create a stack, watch stack events, inspect generated resources, then update or delete the stack from the workbench.

Unsupported resource types are accepted and assigned synthetic physical IDs so broader templates can deploy while you focus on the local services that Floci emulates.

{% elif service.key == 'cognito' %}
About Floci Cognito

Local Cognito stores user pools, app clients, resource servers, users, groups, and tags for AWS-compatible auth workflows. Floci also serves pool-specific discovery and JWKS endpoints for local JWT validation.

Cognito is most useful when testing sign-in, group claims, client credentials, and app code that validates tokens. In Floci, a good learning flow is to create a pool, create a confidential app client, register resource-server scopes, create a user, set a permanent password, then test auth and token validation.

The OAuth token endpoint is emulator-friendly: it supports client_credentials without requiring a Cognito domain and returns local RS256-signed access tokens.

{% elif service.key == 'eventbridge' %}
About Floci EventBridge

Local EventBridge routes application events from event buses through rules and targets such as Lambda, SQS, SNS, and Step Functions.

EventBridge is most often used to decouple event producers from downstream workflows. In Floci, a good learning flow is to select an event bus, inspect the matching rules and targets, send a representative JSON event, then follow the delivered message or invocation in the target service workbench.

Common local test cases include event source naming, detail-type conventions, event pattern matching, disabled rules, target wiring, and app behavior when events are accepted but no rule matches.

{% elif service.key == 'ec2' %}
About Floci EC2

Local EC2 can launch Docker-backed instances, manage lifecycle state, inject imported SSH public keys, run UserData, and expose IMDS-compatible metadata for local compute workflows.

EC2 is most often used to test software that expects an instance-like runtime, VPC placement, security group metadata, or IAM instance profile credentials. In Floci, a good learning flow is to import a key, launch an instance from a built-in AMI, inspect its network placement, then stop, start, reboot, or terminate it from the workbench.

Common local test cases include UserData bootstrapping, instance metadata calls, IAM instance profile credential wiring, security group inventory, and app behavior across stop/start or termination.

{% elif service.key == 'ecr' %}
About Floci ECR

Local ECR combines an AWS-compatible control plane with a real OCI Distribution v2 registry sidecar for Docker push and pull workflows.

ECR is most useful when testing image-backed Lambda functions, ECS task definitions, CDK Docker assets, and repository automation. In Floci, a good learning flow is to create a repository, copy the docker login command, push an image, inspect image tags and digests, then delete images or the repository when the local scenario is done.

Lifecycle and repository policies are stored for API compatibility but are not enforced. Deleted image blobs are reclaimed by the local garbage-collection admin endpoint.

{% elif service.key == 'ecs' %}
About Floci ECS

Local ECS stores clusters, task definitions, tasks, services, task sets, container instances, capacity providers, account settings, attributes, and tags through the AWS-compatible JSON API.

ECS is most useful when testing containerized local workloads and service orchestration. In Floci, a good learning flow is to create a cluster, register a task definition, run a one-off task, then create a service and adjust its desired count.

By default tasks launch as real Docker containers and need Docker socket access. Set FLOCI_SERVICES_ECS_MOCK=true when tests or CI should run without Docker.

{% elif service.key == 'apigateway' %}
About Floci API Gateway

Local API Gateway supports REST APIs and HTTP APIs for testing request routing, integrations, stages, and local application entry points.

API Gateway is most often used to put an HTTP interface in front of Lambda functions, HTTP_PROXY integrations, and service-backed workflows. In Floci, a good learning flow is to select an API, choose a stage and route, send a representative request, then follow the downstream Lambda logs or queue messages.

Common local test cases include method and path matching, stage selection, JSON payload parsing, integration status codes, headers, query strings, and behavior when a route is missing.

{% elif service.key == 'kinesis' %}
About Floci Kinesis

Local Kinesis stores stream records by shard so producers and consumers can test event payloads, partition keys, sequence numbers, and shard iteration behavior.

Kinesis is most often used for ordered event streams, clickstreams, analytics pipelines, and fan-out consumers. In Floci, a good learning flow is to create a stream, put a representative JSON record, then read records from a shard to verify what a consumer will receive.

Common local test cases include partition key choice, malformed payloads, consumer checkpointing, sequence-number handling, and downstream Lambda or EventBridge Pipes integrations.

{% elif service.key == 'kafka' %}
About Floci MSK

Local MSK orchestrates Redpanda containers for Kafka-compatible clusters while keeping the AWS-compatible MSK management API local.

When mock mode is disabled, each created cluster starts a Redpanda container with Kafka port 9092 mapped to a dynamic host port. The cluster becomes ACTIVE after the Redpanda readiness endpoint responds.

A good learning flow is to create a one-broker cluster, refresh until it becomes ACTIVE, load bootstrap brokers, test a Kafka client against the returned localhost endpoint, then delete the cluster when the scenario is done.

{% elif service.key == 'firehose' %}
About Floci Data Firehose

Local Data Firehose accepts records through the AWS-compatible JSON 1.1 API and buffers them for delivery to S3.

Floci flushes buffered records after every five records for quick local feedback. Delivered objects are raw NDJSON in the floci-firehose-results bucket, which makes the S3 workbench a useful next stop after writing test records.

A good learning flow is to create a delivery stream, put one representative JSON record, send a small batch to trigger a flush, then inspect the destination bucket for newline-delimited output.

{% elif service.key == 'secretsmanager' %}
About Floci Secrets Manager

Local Secrets Manager stores application secrets and versions for AWS-compatible development workflows. Read-only inventory keeps values masked, while this workbench can explicitly reveal a selected local value for debugging.

Secrets Manager is most often used for database credentials, API tokens, webhook secrets, and app configuration that should not live in source code. In Floci, a good learning flow is to create a secret, update its value, reload it from the app, then verify the current version and stages.

Common local test cases include missing secrets, malformed JSON values, stale credentials, scheduled deletion, KMS key wiring, and application behavior after secret rotation or update.

{% elif service.key == 'ssm' %}
About Floci SSM Parameter Store

Local SSM includes Parameter Store metadata plus broader SSM inventory. This workbench focuses on parameters for application configuration, feature flags, endpoints, and non-secret runtime settings.

Parameter values stay out of the read-only inventory by default. Use the explicit reveal action when you need to debug what a local app will receive from GetParameter.

Common local test cases include missing parameters, stale config versions, malformed JSON values, SecureString decryption, overwrite behavior, and application reloads after updates.

{% elif service.key == 'rds' %}
About Floci RDS

Local RDS manages real PostgreSQL, MySQL, and MariaDB Docker containers through the AWS-compatible management API, then exposes each database through a localhost TCP proxy port.

RDS is most useful when testing application database connection strings, migrations, startup health checks, and IAM database authentication. In Floci, a good learning flow is to create an instance, wait for its proxy endpoint, connect with psql or mysql, reboot it, then delete it when the local scenario is done.

Each DB instance or cluster gets its own Docker volume. In memory mode volumes are removed on delete; persistent modes retain data for manual recovery unless pruning is enabled.

{% elif service.key == 'backup' %}
About Floci AWS Backup

Local AWS Backup stores backup vaults, plans, selections, simulated backup jobs, recovery points, protected resources, and tags through the AWS-compatible REST JSON API.

Backup jobs move from CREATED to RUNNING and then COMPLETED after the configured completion delay. Completed jobs create recovery points in the target vault, but no underlying resource data is read or written.

A good learning flow is to create a vault, create a plan, add a resource selection, start an on-demand backup job, refresh until it completes, then inspect or delete the generated recovery point.

{% elif service.key == 'transfer' %}
About Floci Transfer Family

Local Transfer Family covers the management-plane API for servers, users, SSH public keys, and tags through the AWS-compatible JSON 1.1 API.

Actual SFTP, FTP, FTPS, and AS2 data-plane file transfer is out of scope. Server state is simulated in-process, with StartServer and StopServer moving servers between ONLINE and OFFLINE.

A good learning flow is to create an SFTP server, create a user with an IAM role and home directory, import an SSH public key, tag the server, then stop and delete the server when testing is complete.

{% elif service.key == 'textract' %}
About Floci Textract

Local Textract emulates the AWS JSON 1.1 API with fixed stub responses. Document bytes and S3 document references are accepted but not parsed.

Every sync response and completed async job returns a synthetic PAGE, LINE, and WORD hierarchy where the extracted text is "Floci" and confidence is 99.9.

Async StartDocumentTextDetection and StartDocumentAnalysis return in-memory job IDs that are immediately SUCCEEDED. Job IDs are not persisted across restarts.

{% elif service.key == 'transcribe' %}
About Floci Transcribe

Local Transcribe is a small AWS-compatible control-plane stub for transcription jobs and custom vocabularies. No real audio parsing or speech-to-text processing is performed.

Jobs transition to COMPLETED immediately and return a synthetic transcript file URI. Vocabularies transition to READY immediately, making both flows useful for local SDK and application control-path testing.

LanguageCode defaults to en-US and MediaFormat defaults to mp4. Job and vocabulary state is in memory and does not persist across Floci restarts.

{% elif service.key == 'bedrockruntime' %}
About Floci Bedrock Runtime

Local Bedrock Runtime emulates Converse and InvokeModel with AWS-compatible fixed stub responses. No real model inference is performed.

Converse requires a non-empty messages array. System prompts, inference configuration, and tool configuration are accepted but ignored; tool-use round-tripping is not implemented.

InvokeModel accepts the request body as opaque bytes and returns an Anthropic-shaped body for Anthropic model IDs or a generic outputs body otherwise. Streaming and the Bedrock management plane are out of scope.

{% elif service.key == 'appconfig' %}
About Floci AppConfig

Local AppConfig supports the management-plane workflow for applications, environments, configuration profiles, hosted versions, deployment strategies, and deployments.

AppConfigData provides the retrieval plane used by applications. Start a configuration session for an application, environment, and profile, then use the returned token to fetch the latest deployed configuration.

A good learning flow is to create an application hierarchy, add a JSON hosted version, create an immediate deployment strategy, deploy the version, then retrieve it through the data plane.

{% elif service.key == 'resourcegroupstagging' %}
About Floci Resource Groups Tagging

Local Resource Groups Tagging provides centralized tag discovery across arbitrary AWS-shaped resource ARNs. Referenced resources do not need to exist in another emulated service.

Searches can be restricted by ARN, tag filters, or service and resource type filters such as lambda or ec2:instance. Global ARNs such as S3 buckets remain visible across regions.

Tags are stored in process and do not persist across Floci restarts. AWS Organizations tag policies and compliance enforcement are out of scope.

{% elif service.key == 'neptune' %}
About Floci Neptune

Local Neptune manages real Apache TinkerPop Gremlin Server containers through the AWS-compatible Query management API.

Each cluster exposes a local Gremlin data-plane endpoint through the configured proxy port range. The first cluster usually claims port 8182, and each additional cluster increments from there.

A good learning flow is to create a cluster, create an instance in it, copy the Gremlin WebSocket endpoint, add a vertex with gremlin-python, then delete the instance and cluster when the graph scenario is done.

{% elif service.key == 'autoscaling' %}
About Floci Auto Scaling

Local Auto Scaling stores launch configurations, groups, lifecycle hooks, policies, target group attachments, and activity history through the AWS-compatible Query API.

Auto Scaling is especially useful in Floci because the capacity reconciler launches and terminates local EC2 instances to maintain desired capacity. A good learning flow is to create a launch configuration, create a group with desired capacity, watch instances move toward InService, then scale out or scale in.

Attached ELB v2 target groups are registered as instances become InService, and scale-in deregisters targets before terminating instances.

{% elif service.key == 'elasticloadbalancing' %}
About Floci ELB v2

Local ELBv2 supports the Application Load Balancer and Network Load Balancer control plane: load balancers, target groups, targets, listeners, listener rules, tags, SSL policy metadata, and account limits.

This is a Phase 1 implementation. The AWS-compatible CRUD APIs are available, but listener data-plane traffic forwarding is planned for a later phase.

A good learning flow is to create a load balancer, create a target group, register local EC2 or IP targets, create a listener with a default forward action, then add a path-based rule.

{% elif service.key == 'eks' %}
About Floci EKS

Local EKS uses the AWS-compatible REST JSON API for cluster metadata, tags, and cluster lifecycle calls.

In mock mode, clusters become ACTIVE immediately and no Kubernetes API server is started. In real mode, Floci starts a k3s container per cluster and exposes the API server on a host port from the configured range.

This first workbench focuses on the supported Phase 1 surface: create clusters, inspect cluster endpoints and metadata, add or remove tags, and delete clusters.

{% elif service.key == 'elasticache' %}
About Floci ElastiCache

Local ElastiCache uses the AWS-compatible Query management API and runs real Valkey or Redis Docker containers behind TCP proxy ports.

A good learning flow is to create a replication group, inspect the primary endpoint port, connect with redis-cli or your application, then delete the group when the local cache is no longer needed.

Floci also supports ElastiCache users and IAM auth token validation, so local cache clients can test RBAC-style access strings and IAM-authenticated connection flows.

{% elif service.key == 'opensearch' %}
About Floci OpenSearch

Local OpenSearch uses the AWS-compatible REST JSON management API. In mock mode, domain metadata is stored in-process; in real mode, Floci starts one OpenSearch or Elasticsearch container per domain.

A good learning flow is to create a domain, inspect its endpoint and health, test config changes, add tags, then delete the domain when the local search cluster is no longer needed.

Floci validates supported engine versions and exposes instance type details so SDK clients and infrastructure tools can inspect realistic domain metadata while the runtime stays local.

{% elif service.key == 'athena' %}
About Floci Athena

Local Athena uses the AWS-compatible JSON 1.1 API and runs SQL through a floci-duck sidecar powered by DuckDB.

A good learning flow is to register S3-backed tables in Glue, start a SQL query here, watch the query state, then fetch the completed result rows from the S3-backed CSV output.

Mock mode skips DuckDB execution and immediately returns successful empty results, which is useful when tests only need the Athena query state machine.

{% elif service.key == 'glue' %}
About Floci Glue

Local Glue emulates the Data Catalog and Schema Registry so local data lake metadata, schema compatibility workflows, and Athena table resolution can be tested through AWS-compatible APIs.

A good learning flow is to create a database, register an S3-backed JSON or Parquet table, add partitions when needed, then query the table from Athena. The dashboard infers which DuckDB reader Athena will use from the table's input format and SerDe library.

The Schema Registry flow supports AVRO, JSON, and PROTOBUF definitions with the compatibility modes exposed by Floci, making it useful for testing event producer and consumer schema-version workflows.

{% elif service.key == 'codebuild' %}
About Floci CodeBuild

Local CodeBuild stores project, report-group, and source-credential configuration and executes builds inside real Docker containers.

Buildspec install, pre_build, build, and post_build phases run sequentially. Output is streamed to CloudWatch Logs under /aws/codebuild/<project>, and configured S3 artifacts are copied from the container after the build.

A good learning flow is to create a NO_SOURCE project with a lightweight image, start a build using an inline buildspec override, poll until buildComplete, then inspect its phases, logs, and artifacts.

{% elif service.key == 'codedeploy' %}
About Floci CodeDeploy

Local CodeDeploy stores applications, deployment groups, deployment configs, deployments, tags, and on-premises no-op metadata through the AWS-compatible JSON 1.1 API.

For Lambda and ECS compute platforms, Floci performs real local deployment execution: Lambda alias traffic shifting or ECS blue/green task-set and ELB listener updates, including lifecycle hook callbacks.

A good learning flow is to create a Lambda or ECS application, create a deployment group, submit an AppSpecContent revision, then poll the deployment and inspect deployment targets or hook status.

{% elif service.key == 'cloudfront' %}
About Floci CloudFront

Local CloudFront emulates the management plane for distributions, invalidations, cache policies, origin request policies, response headers policies, origin access identities, origin access controls, functions, and tags.

Actual edge delivery is not emulated. This workbench focuses on creating and inspecting CDN configuration so local infrastructure, IaC flows, and SDK calls can be tested without AWS.

A good learning flow is to create a distribution for an S3-style origin, update its enabled state or comment, create a cache invalidation, then inspect the generated domain name and global ARN.

{% elif service.key == 'route53' %}
About Floci Route 53

Local Route 53 emulates the management plane for hosted zones, record-set changes, health checks, change tracking, and tags.

Actual DNS resolution is not provided. This workbench focuses on creating and inspecting DNS configuration so local infrastructure and SDK flows can be tested without AWS.

A good learning flow is to create a hosted zone, add an A or CNAME record, inspect the immediate INSYNC change, create a health check, then tag the zone or health check.

{% elif service.key == 'scheduler' %}
About Floci EventBridge Scheduler

Local EventBridge Scheduler stores schedule groups and schedules through the AWS-compatible REST JSON API. When invocation is enabled, a local dispatcher fires due schedules on a short polling interval.

Scheduler is most often used to trigger Lambda functions, send SQS or SNS messages, or put EventBridge events on a recurring or one-time cadence. In Floci, a good learning flow is to create a group, add a rate or at expression, point the target at a local service, then watch the target service workbench for the fired payload.

Supported expressions include at, rate, and six-field cron. Retry policy and dead-letter config are stored for compatibility, but failed-invocation retry behavior is not enforced yet.

{% elif service.key == 'pipes' %}
About Floci EventBridge Pipes

Local EventBridge Pipes stores pipe definitions that connect supported sources to supported targets, with optional enrichment and parameter blocks for service-specific behavior.

Pipes are most useful for testing local event handoffs such as SQS to Lambda, Kinesis to Step Functions, or DynamoDB stream records to SNS. In Floci, a good learning flow is to create a pipe with source and target ARNs, start it, then inspect the downstream workbench for delivered events.

Supported sources include SQS queues, Kinesis streams, DynamoDB streams, and Kafka topics. Supported targets include Lambda, SQS, SNS, Kinesis, and Step Functions.

{% elif service.key == 'config' %}
About Floci AWS Config

Local AWS Config stores configuration recorders, delivery channels, config rules, conformance packs, tags, and compliance metadata through the AWS-compatible JSON 1.1 API.

Config is most useful for testing local compliance setup and infrastructure automation. In Floci, a good learning flow is to create a recorder, add a delivery channel, start recording, create a rule, then trigger a rule evaluation.

Compliance status always returns INSUFFICIENT_DATA because Floci stores the management-plane resources but does not perform real resource evaluation or configuration recording.
