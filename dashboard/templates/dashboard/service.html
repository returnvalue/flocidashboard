{% if service.key == 's3' %}
About Floci S3

Local S3 emulator. Bucket policies and lifecycle rules are stored but not enforced at access time. Replication, access logging, and inventory are not available.

S3 is most often used as durable object storage for uploads, generated files, backups, static assets, data lake inputs, and service-to-service handoffs. In Floci, a good learning flow is to create a bucket, upload a file, organize objects with folder-style prefixes, inspect metadata and tags, then generate a presigned URL to test temporary access.

Common local test cases include upload and download paths, overwrite behavior, empty bucket cleanup, versioning changes, object tagging, copy operations, prefix browsing, and app behavior when a bucket or object does not exist.

{% elif service.key == 'iam' %}
About Floci IAM

Local IAM stores users, roles, groups, policies, access keys, and trust relationships for AWS-compatible development workflows.

IAM is most often used to shape what local SDK calls are allowed to do and to test role-based credentials. In Floci, a good learning flow is to inspect a principal, review its attached and inline policies, create a temporary access key or assume a role, then test those credentials from your app.

Common local test cases include missing permissions, incorrect trust policies, inactive access keys, group policy inheritance, permission boundaries, and role-session credential wiring.

{% elif service.key == 'sqs' %}
About Floci SQS

Local SQS emulator for queue-driven workflows. Polling messages makes them temporarily invisible until the visibility timeout expires or they are deleted.

SQS is most often used to decouple producers from background workers, smooth traffic spikes, retry failed work, and connect services without requiring them to run at the same time. In Floci, a good learning flow is to create a queue, send a test message, poll it from the UI, inspect the body and attributes, then delete it after your worker logic succeeds.

Common local test cases include producer message format, consumer retry behavior, visibility timeout handling, delete-after-success behavior, purge cleanup, FIFO ordering, duplicate handling, and dead-letter queue flows.

{% elif service.key == 'sns' %}
About Floci SNS

Local SNS emulator for topic-based pub/sub workflows. Published messages are delivered immediately to supported local subscriptions, including SQS queues.

SNS is most often used to fan out events to queues, functions, webhooks, and notification handlers. In Floci, a good learning flow is to create a topic with your app or CLI, subscribe an SQS queue, publish a test event here, then inspect the delivered message in the SQS workbench.

Common local test cases include message shape, attributes, subject handling, subscription filter policies, FIFO group and deduplication behavior, and SNS-to-SQS fanout.

{% elif service.key == 'lambda' %}
About Floci Lambda

Local Lambda runs function code in Docker-backed execution environments. Invocations can return response payloads, function errors, and a tail of execution logs.

Lambda is most often used to handle API Gateway requests, consume SQS messages, react to S3 or EventBridge events, and run small background tasks. In Floci, a good learning flow is to select a function, invoke it with representative JSON, inspect the payload and log tail, then follow the related CloudWatch log group.

Common local test cases include handler payload parsing, environment variables, timeout behavior, function errors, event-source integration payloads, and log output.

{% elif service.key == 'dynamodb' %}
About Floci DynamoDB

Local DynamoDB stores application state in tables with AWS-compatible key, index, stream, and item APIs. PartiQL SELECT support is available in Floci 1.5.17 and newer.

DynamoDB is most often used for low-latency application records, counters, job state, idempotency keys, and event projections. In Floci, a good learning flow is to inspect a table schema, scan a bounded set of items, select an item for JSON detail, then try a read-only PartiQL SELECT query.

Common local test cases include key design, item shape, stream enablement, TTL configuration, index projections, and app behavior when records are missing or malformed.

{% elif service.key == 'cloudwatch' %}
About Floci CloudWatch Logs

Local CloudWatch Logs captures log groups, log streams, and log events produced by services such as Lambda. This viewer focuses on recent logs for local debugging rather than the full CloudWatch Console surface.

A good learning flow is to select a Lambda log group, open the newest stream, refresh recent events, and compare the messages with the request or event that triggered the function.

Common local test cases include handler errors, async event processing, retry behavior, missing environment configuration, and log messages emitted by background workers.

{% elif service.key == 'stepfunctions' %}
About Floci Step Functions

Local Step Functions runs state machine workflows through the AWS-compatible Amazon States API. This workbench focuses on starting executions and inspecting their status, input, output, and recent history.

Step Functions is most often used to coordinate Lambda functions, service integrations, retries, waits, and branching workflows. In Floci, a good learning flow is to select a state machine, start an execution with representative JSON, inspect the event history, then stop a running execution when testing cancellation behavior.

Common local test cases include input shape, Choice state routing, Lambda task payloads, retry and catch paths, timeout behavior, failure output, and task-token callback flows.

{% elif service.key == 'cloudformation' %}
About Floci CloudFormation

Local CloudFormation deploys templates through the AWS-compatible Query XML API and provisions supported local resources such as S3 buckets, SQS queues, SNS topics, DynamoDB tables, Lambda functions, IAM identities, SSM parameters, KMS keys, Secrets Manager secrets, API Gateway resources, EventBridge rules, and more.

CloudFormation is most useful when testing whole local application slices. In Floci, a good learning flow is to validate a template, create a stack, watch stack events, inspect generated resources, then update or delete the stack from the workbench.

Unsupported resource types are accepted and assigned synthetic physical IDs so broader templates can deploy while you focus on the local services that Floci emulates.

{% elif service.key == 'cognito' %}
About Floci Cognito

Local Cognito stores user pools, app clients, resource servers, users, groups, and tags for AWS-compatible auth workflows. Floci also serves pool-specific discovery and JWKS endpoints for local JWT validation.

Cognito is most useful when testing sign-in, group claims, client credentials, and app code that validates tokens. In Floci, a good learning flow is to create a pool, create a confidential app client, register resource-server scopes, create a user, set a permanent password, then test auth and token validation.

The OAuth token endpoint is emulator-friendly: it supports client_credentials without requiring a Cognito domain and returns local RS256-signed access tokens.

{% elif service.key == 'eventbridge' %}
About Floci EventBridge

Local EventBridge routes application events from event buses through rules and targets such as Lambda, SQS, SNS, and Step Functions.

EventBridge is most often used to decouple event producers from downstream workflows. In Floci, a good learning flow is to select an event bus, inspect the matching rules and targets, send a representative JSON event, then follow the delivered message or invocation in the target service workbench.

Common local test cases include event source naming, detail-type conventions, event pattern matching, disabled rules, target wiring, and app behavior when events are accepted but no rule matches.

{% elif service.key == 'ec2' %}
About Floci EC2

Local EC2 can launch Docker-backed instances, manage lifecycle state, inject imported SSH public keys, run UserData, and expose IMDS-compatible metadata for local compute workflows.

EC2 is most often used to test software that expects an instance-like runtime, VPC placement, security group metadata, or IAM instance profile credentials. In Floci, a good learning flow is to import a key, launch an instance from a built-in AMI, inspect its network placement, then stop, start, reboot, or terminate it from the workbench.

Common local test cases include UserData bootstrapping, instance metadata calls, IAM instance profile credential wiring, security group inventory, and app behavior across stop/start or termination.

{% elif service.key == 'apigateway' %}
About Floci API Gateway

Local API Gateway supports REST APIs and HTTP APIs for testing request routing, integrations, stages, and local application entry points.

API Gateway is most often used to put an HTTP interface in front of Lambda functions, HTTP_PROXY integrations, and service-backed workflows. In Floci, a good learning flow is to select an API, choose a stage and route, send a representative request, then follow the downstream Lambda logs or queue messages.

Common local test cases include method and path matching, stage selection, JSON payload parsing, integration status codes, headers, query strings, and behavior when a route is missing.

{% elif service.key == 'kinesis' %}
About Floci Kinesis

Local Kinesis stores stream records by shard so producers and consumers can test event payloads, partition keys, sequence numbers, and shard iteration behavior.

Kinesis is most often used for ordered event streams, clickstreams, analytics pipelines, and fan-out consumers. In Floci, a good learning flow is to create a stream, put a representative JSON record, then read records from a shard to verify what a consumer will receive.

Common local test cases include partition key choice, malformed payloads, consumer checkpointing, sequence-number handling, and downstream Lambda or EventBridge Pipes integrations.

{% elif service.key == 'secretsmanager' %}
About Floci Secrets Manager

Local Secrets Manager stores application secrets and versions for AWS-compatible development workflows. Read-only inventory keeps values masked, while this workbench can explicitly reveal a selected local value for debugging.

Secrets Manager is most often used for database credentials, API tokens, webhook secrets, and app configuration that should not live in source code. In Floci, a good learning flow is to create a secret, update its value, reload it from the app, then verify the current version and stages.

Common local test cases include missing secrets, malformed JSON values, stale credentials, scheduled deletion, KMS key wiring, and application behavior after secret rotation or update.

{% elif service.key == 'ssm' %}
About Floci SSM Parameter Store

Local SSM includes Parameter Store metadata plus broader SSM inventory. This workbench focuses on parameters for application configuration, feature flags, endpoints, and non-secret runtime settings.

Parameter values stay out of the read-only inventory by default. Use the explicit reveal action when you need to debug what a local app will receive from GetParameter.

Common local test cases include missing parameters, stale config versions, malformed JSON values, SecureString decryption, overwrite behavior, and application reloads after updates.
